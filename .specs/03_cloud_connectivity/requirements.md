# Requirements Document: CLOUD_GATEWAY + CLOUD_GATEWAY_CLIENT + Mock COMPANION_APP

## Introduction

This specification defines the vehicle-to-cloud connectivity layer: the
CLOUD_GATEWAY backend service (Go) that exposes a REST API and communicates
with vehicles via MQTT, the CLOUD_GATEWAY_CLIENT service (Rust) that runs in
the RHIVOS safety partition and bridges MQTT to the Kuksa DATA_BROKER, and
the mock COMPANION_APP CLI tool for testing. Together these enable remote
lock/unlock commands and vehicle telemetry.

## Glossary

| Term | Definition |
|------|-----------|
| Bearer token | An opaque authentication token obtained through vehicle pairing |
| CLOUD_GATEWAY | Go backend service providing REST API and MQTT routing |
| CLOUD_GATEWAY_CLIENT | Rust service in RHIVOS safety partition bridging MQTT to DATA_BROKER |
| Command correlation | Matching a REST command to its MQTT response using VIN and command_id |
| COMPANION_APP | Mobile application for remote vehicle control (mocked by CLI in this spec) |
| DATA_BROKER | Eclipse Kuksa Databroker (configured in spec 02) |
| MQTT | Message Queuing Telemetry Transport protocol |
| Mosquitto | Eclipse Mosquitto MQTT broker (infrastructure from spec 01) |
| Pairing | Process of associating a COMPANION_APP instance with a specific vehicle VIN |
| Pairing PIN | A 6-digit numeric code generated by CLOUD_GATEWAY_CLIENT for vehicle pairing |
| QoS | MQTT Quality of Service level (0 = fire-and-forget, 2 = exactly-once) |
| Telemetry | Periodic vehicle state data published by CLOUD_GATEWAY_CLIENT |

## Requirements

### Requirement 1: CLOUD_GATEWAY REST API

**User Story:** As a companion app user, I want a REST API to remotely
lock/unlock my vehicle and check its status, so that I can control my vehicle
from my phone.

#### Acceptance Criteria

1. **03-REQ-1.1** THE CLOUD_GATEWAY SHALL expose `POST /api/v1/vehicles/{vin}/lock`
   that accepts a lock command and returns `202 Accepted` with a JSON body
   containing a `command_id`.

2. **03-REQ-1.2** THE CLOUD_GATEWAY SHALL expose `POST /api/v1/vehicles/{vin}/unlock`
   with the same behavior as the lock endpoint.

3. **03-REQ-1.3** THE CLOUD_GATEWAY SHALL expose `GET /api/v1/vehicles/{vin}/status`
   that returns the latest known vehicle state (lock state, door state, speed,
   location, parking session).

4. **03-REQ-1.4** THE CLOUD_GATEWAY SHALL expose `GET /healthz` that returns
   `200 OK` when the service is operational.

5. **03-REQ-1.5** WHEN a lock or unlock endpoint is called, THE CLOUD_GATEWAY
   SHALL NOT block the REST response on the MQTT round-trip. The response
   SHALL be returned immediately with status `202 Accepted`.

6. **03-REQ-1.6** THE CLOUD_GATEWAY SHALL accept a listen address via
   `--listen-addr` flag or `LISTEN_ADDR` environment variable, defaulting to
   `:8081`.

#### Edge Cases

1. **03-REQ-1.E1** IF a command is sent for an unregistered VIN, THEN
   CLOUD_GATEWAY SHALL respond with `404 Not Found`.

2. **03-REQ-1.E2** IF a command is sent without a valid bearer token, THEN
   CLOUD_GATEWAY SHALL respond with `401 Unauthorized`.

3. **03-REQ-1.E3** IF the MQTT broker is unreachable when publishing a command,
   THEN CLOUD_GATEWAY SHALL respond with `503 Service Unavailable`.

### Requirement 2: CLOUD_GATEWAY MQTT Integration

**User Story:** As a system component, I want CLOUD_GATEWAY to communicate
with vehicles via MQTT, so that commands reach the vehicle and responses
come back.

#### Acceptance Criteria

1. **03-REQ-2.1** WHEN CLOUD_GATEWAY starts, THE service SHALL connect to the
   MQTT broker (Mosquitto) and subscribe to `vehicles/+/command_responses`
   (QoS 2), `vehicles/+/status_response` (QoS 2), `vehicles/+/telemetry`
   (QoS 0), and `vehicles/+/registration` (QoS 2).

2. **03-REQ-2.2** WHEN a lock or unlock command is accepted via REST, THE
   CLOUD_GATEWAY SHALL publish a command message to
   `vehicles/{vin}/commands` with QoS 2.

3. **03-REQ-2.3** WHEN a command response is received via MQTT, THE
   CLOUD_GATEWAY SHALL update the vehicle's internal state with the command
   result.

4. **03-REQ-2.4** WHEN a telemetry message is received via MQTT, THE
   CLOUD_GATEWAY SHALL update the vehicle's cached state.

5. **03-REQ-2.5** THE CLOUD_GATEWAY SHALL accept an MQTT broker address via
   `--mqtt-addr` flag or `MQTT_ADDR` environment variable, defaulting to
   `localhost:1883`.

#### Edge Cases

1. **03-REQ-2.E1** IF the MQTT connection is lost, THEN CLOUD_GATEWAY SHALL
   reconnect automatically with exponential backoff.

2. **03-REQ-2.E2** IF a command response references an unknown `command_id`,
   THEN CLOUD_GATEWAY SHALL log a warning and discard the message.

### Requirement 3: CLOUD_GATEWAY_CLIENT Command Processing

**User Story:** As a vehicle system, I want CLOUD_GATEWAY_CLIENT to receive
remote commands via MQTT and execute them through DATA_BROKER, so that the
companion app can control vehicle functions.

#### Acceptance Criteria

1. **03-REQ-3.1** WHEN CLOUD_GATEWAY_CLIENT starts, THE service SHALL connect
   to the MQTT broker and subscribe to `vehicles/{vin}/commands` (QoS 2) and
   `vehicles/{vin}/status_request` (QoS 2).

2. **03-REQ-3.2** WHEN a lock command is received via MQTT, THE
   CLOUD_GATEWAY_CLIENT SHALL write `Vehicle.Command.Door.Lock = true` to
   DATA_BROKER.

3. **03-REQ-3.3** WHEN an unlock command is received via MQTT, THE
   CLOUD_GATEWAY_CLIENT SHALL write `Vehicle.Command.Door.Lock = false` to
   DATA_BROKER.

4. **03-REQ-3.4** THE CLOUD_GATEWAY_CLIENT SHALL subscribe to
   `Vehicle.Command.Door.LockResult` on DATA_BROKER. WHEN a result is
   received, THE service SHALL publish a command response to
   `vehicles/{vin}/command_responses` (QoS 2) containing the `command_id`
   and result.

5. **03-REQ-3.5** WHEN a status request is received via MQTT, THE
   CLOUD_GATEWAY_CLIENT SHALL read current vehicle state from DATA_BROKER
   and publish a status response to `vehicles/{vin}/status_response` (QoS 2).

6. **03-REQ-3.6** THE CLOUD_GATEWAY_CLIENT SHALL accept a DATA_BROKER
   address via `--databroker-addr` flag or `DATABROKER_ADDR` environment
   variable (default: `localhost:55555`) and an MQTT broker address via
   `--mqtt-addr` flag or `MQTT_ADDR` environment variable (default:
   `localhost:1883`).

#### Edge Cases

1. **03-REQ-3.E1** IF the DATA_BROKER is unreachable, THEN
   CLOUD_GATEWAY_CLIENT SHALL retry with exponential backoff and log errors.

2. **03-REQ-3.E2** IF the MQTT connection is lost, THEN
   CLOUD_GATEWAY_CLIENT SHALL reconnect and re-subscribe automatically.

3. **03-REQ-3.E3** IF a received MQTT command has an invalid JSON structure,
   THEN CLOUD_GATEWAY_CLIENT SHALL log the error and discard the message.

### Requirement 4: CLOUD_GATEWAY_CLIENT Telemetry

**User Story:** As a fleet operator (or companion app user), I want the
vehicle to periodically report its state, so that the cloud has an up-to-date
view of vehicle status.

#### Acceptance Criteria

1. **03-REQ-4.1** WHILE CLOUD_GATEWAY_CLIENT is running, THE service SHALL
   periodically read vehicle signals from DATA_BROKER and publish a telemetry
   message to `vehicles/{vin}/telemetry` (QoS 0).

2. **03-REQ-4.2** THE telemetry message SHALL include: `is_locked`,
   `is_door_open`, `speed`, `latitude`, `longitude`,
   `parking_session_active`, `vin`, and `timestamp`.

3. **03-REQ-4.3** THE telemetry publishing interval SHALL be configurable via
   `--telemetry-interval` flag or `TELEMETRY_INTERVAL` environment variable,
   defaulting to 5 seconds.

#### Edge Cases

1. **03-REQ-4.E1** IF a signal value is unavailable in DATA_BROKER, THEN
   the telemetry message SHALL omit that field or use a null/default value.

### Requirement 5: Vehicle Registration and Pairing

**User Story:** As a vehicle owner, I want to pair my companion app with my
vehicle using a PIN displayed on the vehicle, so that only authorized users
can control the vehicle remotely.

#### Acceptance Criteria

1. **03-REQ-5.1** WHEN CLOUD_GATEWAY_CLIENT starts for the first time, THE
   service SHALL generate a VIN and a 6-digit pairing PIN, persist both to
   a configurable data directory, and log them to stdout.

2. **03-REQ-5.2** WHEN CLOUD_GATEWAY_CLIENT starts (first time or subsequent),
   THE service SHALL publish a registration message to
   `vehicles/{vin}/registration` (QoS 2) containing VIN and pairing PIN.

3. **03-REQ-5.3** WHEN CLOUD_GATEWAY receives a registration message, THE
   service SHALL store the VIN and pairing PIN in its vehicle registry.

4. **03-REQ-5.4** THE CLOUD_GATEWAY SHALL expose `POST /api/v1/pair` that
   accepts `{vin, pin}`. WHEN the VIN and PIN match a registered vehicle,
   THE service SHALL generate a bearer token, store it, and return
   `{token, vin}`.

5. **03-REQ-5.5** WHEN a REST endpoint (lock, unlock, status) receives a
   request, THE CLOUD_GATEWAY SHALL validate the `Authorization: Bearer`
   header against the pairing database. The token must be associated with
   the VIN in the request path.

#### Edge Cases

1. **03-REQ-5.E1** IF the VIN in a pair request is not registered, THEN
   CLOUD_GATEWAY SHALL respond with `404 Not Found`.

2. **03-REQ-5.E2** IF the PIN does not match the registered vehicle, THEN
   CLOUD_GATEWAY SHALL respond with `403 Forbidden`.

3. **03-REQ-5.E3** IF CLOUD_GATEWAY_CLIENT starts with an existing VIN file,
   THEN the service SHALL reuse the persisted VIN and PIN (not generate new
   ones).

### Requirement 6: Mock COMPANION_APP CLI

**User Story:** As a developer, I want a CLI tool that simulates the
companion app, so that I can test the full lock/unlock flow without a real
mobile application.

#### Acceptance Criteria

1. **03-REQ-6.1** THE mock `companion-app-cli` SHALL implement a `pair`
   subcommand that calls `POST /api/v1/pair` with `{vin, pin}` and prints
   the returned token.

2. **03-REQ-6.2** THE mock `companion-app-cli` SHALL implement a `lock`
   subcommand that calls `POST /api/v1/vehicles/{vin}/lock` with the bearer
   token and prints the response.

3. **03-REQ-6.3** THE mock `companion-app-cli` SHALL implement an `unlock`
   subcommand that calls `POST /api/v1/vehicles/{vin}/unlock` with the
   bearer token and prints the response.

4. **03-REQ-6.4** THE mock `companion-app-cli` SHALL implement a `status`
   subcommand that calls `GET /api/v1/vehicles/{vin}/status` with the bearer
   token and prints the vehicle state.

5. **03-REQ-6.5** THE mock CLI SHALL accept flags: `--gateway-addr`
   (default: `http://localhost:8081`), `--vin` (required), `--token`
   (required for lock/unlock/status), `--pin` (required for pair).

#### Edge Cases

1. **03-REQ-6.E1** IF the CLOUD_GATEWAY is unreachable, THEN the CLI SHALL
   print an error and exit with a non-zero exit code.
